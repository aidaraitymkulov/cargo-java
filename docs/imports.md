# Импорт Excel (Imports)

## Обзор

Импорт — единственный способ создавать товары и заказы в системе. Три эндпоинта соответствуют трём этапам логистики: регистрация в Китае, приёмка в КР, выдача клиенту.

Все три принимают файл `.xlsx` как `multipart/form-data`, поле `file`.

**Принцип частичного успеха:** ошибка в одной строке/блоке не останавливает весь импорт — обрабатывается всё что можно, ошибки собираются в `errors[]`.

---

## Права доступа

| Роль | CN-импорт | KG-импорт |
|---|---|---|
| `MANAGER` | Любой файл (CN не привязан к филиалу) | Только файлы своего филиала |
| `SUPER_ADMIN` | Любой | Любой филиал |

---

## Формат ответа (все три эндпоинта)

```json
{
  "imported": 42,
  "errors": [
    {
      "row": 7,
      "reason": "USER_NOT_FOUND",
      "detail": "personalCode AN9999 не найден"
    }
  ]
}
```

| Поле | Описание |
|---|---|
| `imported` | Количество успешно обработанных строк/блоков |
| `errors` | Список ошибок — строки/блоки, которые были пропущены |
| `errors[].row` | Номер строки в Excel (1-based) |
| `errors[].reason` | Код ошибки (`USER_NOT_FOUND`, `IMPORT_ERROR` и т.д.) |
| `errors[].detail` | Человекочитаемое описание |

## 1. Импорт из Китая

**`POST /admin/import/parcels/cn`**

Регистрирует товары прибывшие со склада в Китае. Каждая строка = один товар со статусом `IN_CHINA`.

### Формат файла

| Колонка | Содержимое |
|---|---|
| A | `hatch` — трекинг-номер посылки |
| B | `personalCode` — код клиента |

```
YT123456789 | AN0001
JT987654321 | D0015
YT111222333 | AN0001   ← тот же клиент, другой hatch — это нормально
```

Строки где `hatch` или `personalCode` пустые — молча пропускаются.

### Алгоритм (на каждую строку)

1. Найти пользователя по `personalCode` → ошибка `USER_NOT_FOUND` если не найден
2. Создать `Product` со статусом `IN_CHINA`
3. Записать в `product_histories`

### Особенности

- CN-файл общий для всех филиалов — ограничений по филиалу нет ни для `MANAGER`, ни для `SUPER_ADMIN`
- Дубликаты `hatch` у одного клиента — норма, создаётся отдельный товар
- Каждая строка — отдельная транзакция; ошибка одной строки не откатывает остальные

## 2. Импорт после сортировки (in-kg)

**`POST /admin/import/parcels/kg/{prefix}/in-kg`**

Регистрирует прибытие посылок в КР и создаёт заказы. `{prefix}` — код филиала (`AN`, `D`, `OW`).

### Формат файла (блочный)

Файл разбит на блоки — один блок на одного клиента. Блок заканчивается строкой с `personalCode`.

| Колонка | Содержимое |
|---|---|
| A | `personalCode` — **только на итоговой строке блока** |
| B | `hatch` — на каждой строке блока, включая итоговую |
| C | `weight` — **только на итоговой строке** (кг, десятичное) |
| D | `price` — **только на итоговой строке** (сом, десятичное) |

```
        | YT123456789 |     |
        | JT987654321 |     |
AN0001  | YT111222333 | 3.5 | 4500   ← итоговая строка
        | YT555666777 |     |
D0015   | YT999888111 | 1.2 | 1800   ← итоговая строка следующего блока
```

Блок пропускается (с предупреждением в лог) если: пустой `weight`/`price`, или нет ни одного `hatch`.

### Алгоритм (на каждый блок)

1. Найти пользователя по `personalCode`
2. Создать `Order` (статус `PENDING_PICKUP`, `weight`, `price`, `itemCount = hatches.size()`)
3. Для каждого `hatch` в блоке:
   - Найти товар: самый старый непривязанный (`order_id IS NULL`) товар этого клиента с этим `hatch` и статусом не `DELIVERED` — **FIFO по `created_at`**
   - Если нашли: привязать к заказу, перевести в `IN_KG`
   - Если не нашли: создать новый товар сразу с `IN_KG`
4. Записать в `product_histories` для каждого товара

### Транзакции

Каждый блок обрабатывается в **отдельной транзакции** (`REQUIRES_NEW`). Ошибка одного блока не откатывает уже обработанные. Это реализовано через отдельный бин `KgBlockProcessor` — так Spring-прокси корректно создаёт новую транзакцию.

## 3. Импорт выдачи (delivered)

**`POST /admin/import/parcels/kg/{prefix}/delivered`**

Фиксирует выдачу товаров клиентам. Формат файла **идентичен** `in-kg`.

### Алгоритм (на каждый блок)

1. Найти пользователя по `personalCode`
2. Найти существующий заказ: среди `PENDING_PICKUP`-заказов пользователя этого филиала найти тот, чей набор `hatch` совпадает с блоком (сравнение по множеству `Set<String>`)
3. Проверить что заказ ещё не выдан → ошибка `ORDER_ALREADY_DELIVERED` если уже `DELIVERED`
4. Перевести заказ в `DELIVERED`
5. Все товары заказа перевести в `DELIVERED`
6. Записать в `product_histories`

### Поиск заказа

Заказ ищется по точному совпадению набора `hatch` — не по подмножеству, а по равенству множеств. Если совпадения нет → `ORDER_NOT_FOUND`.

---

## Парсер Excel

Используется Apache POI (`XSSFWorkbook`). Обрабатывает ячейки трёх типов:
- `NUMERIC` — числа (hatch иногда хранятся как числа в Excel), форматируются без дробной части если целые
- `STRING` — строки
- `FORMULA` — вычисляются через кэшированное значение

Разделитель дробной части: и `.` и `,` принимаются для `weight`/`price`.

## Типичные ошибки импорта

| Код | Причина |
|---|---|
| `USER_NOT_FOUND` | `personalCode` не найден в системе |
| `BRANCH_NOT_FOUND` | Филиал с указанным `prefix` не существует |
| `FORBIDDEN` | Менеджер пытается загрузить файл чужого филиала |
| `ORDER_NOT_FOUND` | При `delivered` — заказ с таким набором hatch не найден |
| `ORDER_ALREADY_DELIVERED` | Заказ уже был выдан ранее |
| `IMPORT_ERROR` | Не удалось прочитать файл Excel |
